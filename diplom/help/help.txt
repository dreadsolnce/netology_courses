Для новичка процесс деплоя из GitLab в Yandex Cloud (YC) Kubernetes (Managed Service for Kubernetes) лучше всего разбить на понятные этапы. Мы будем использовать современный и безопасный метод — GitLab Agent for Kubernetes.

  Вот пошаговый план:

  ---


  1. Подготовка в Yandex Cloud
  Прежде всего, у вас должны быть созданы:
   1. Кластер Kubernetes в консоли Yandex Cloud.
   2. Container Registry (реестр), куда мы будем складывать Docker-образы.
   3. Сервисный аккаунт (например, gitlab-sa) с ролями:
       * container-registry.images.pusher (чтобы загружать образы).
       * k8s.clusters.agent (для работы агента GitLab).


  2. Связка GitLab и Kubernetes (GitLab Agent)
  Раньше использовали сертификаты и токены, но сейчас стандарт — GitLab Agent. Это маленькая программка, которая ставится внутрь вашего кластера и "слушает" команды из GitLab.


   1. В вашем проекте GitLab перейдите в Operate -> Kubernetes clusters.
   2. Нажмите Connect a cluster и выберите Create a new agent.
   3. Дайте ему имя (например, yc-agent).
   4. GitLab выдаст вам команду helm install....
   5. Выполните эту команду через терминал (у вас должен быть настроен kubectl для доступа к кластеру YC).


  Теперь GitLab "видит" ваш кластер через этот агент.

  ---


  3. Настройка переменных в GitLab
  Чтобы GitLab мог собрать образ и отправить его в Yandex Cloud, ему нужны доступы.
  Перейдите в Settings -> CI/CD -> Variables и добавьте:
   * YC_REGISTRY_ID: ID вашего реестра из консоли YC.
   * YC_SA_KEY: JSON-ключ вашего сервисного аккаунта (его можно создать в консоли YC: Сервисный аккаунт -> Создать новый ключ).

  ---


  4. Создание Docker-образа (Pipeline)
  Создайте в корне проекта файл .gitlab-ci.yml. Он описывает, что делать.


    1 stages:
    2   - build
    3   - deploy
    4
    5 build-image:
    6   stage: build
    7   image: docker:24.0.5
    8   services:
    9     - docker:24.0.5-dind
   10   script:
   11     # 1. Авторизация в Yandex Container Registry
   12     - echo "$YC_SA_KEY" | docker login --username json_key --password-stdin cr.yandex
   13     # 2. Сборка образа
   14     - docker build -t cr.yandex/$YC_REGISTRY_ID/my-app:$CI_COMMIT_SHORT_SHA .
   15     # 3. Пуш (отправка) в облако
   16     - docker push cr.yandex/$YC_REGISTRY_ID/my-app:$CI_COMMIT_SHORT_SHA
   17
   18 deploy-to-k8s:
   19   stage: deploy
   20   image:
   21     name: bitnami/kubectl:latest
   22     entrypoint: [""]
   23   script:
   24     # Указываем контекст агента (замените путь на ваш: <группа>/<проект>:<имя_агента>)
   25     - kubectl config use-context path/to/your/project:yc-agent
   26     # Подставляем ID реестра и версию образа в манифесты и применяем их
   27     - sed -i "s|__IMAGE__|cr.yandex/$YC_REGISTRY_ID/my-app:$CI_COMMIT_SHORT_SHA|g" k8s/deployment.yaml
   28     - kubectl apply -f k8s/

  ---


  5. Манифесты Kubernetes
  В папке k8s/ вашего проекта должны лежать файлы, описывающие приложение.

  Файл k8s/deployment.yaml:


    1 apiVersion: apps/v1
    2 kind: Deployment
    3 metadata:
    4   name: my-app
    5 spec:
    6   replicas: 2
    7   selector:
    8     matchLabels:
    9       app: my-app
   10   template:
   11     metadata:
   12       labels:
   13         app: my-app
   14     spec:
   15       containers:
   16       - name: my-app
   17         image: __IMAGE__  # Эту строку заменит скрипт в GitLab CI
   18         ports:
   19         - containerPort: 80
   20 ---
   21 apiVersion: v1
   22 kind: Service
   23 metadata:
   24   name: my-app-service
   25 spec:
   26   type: LoadBalancer # Создаст внешний IP в Yandex Cloud
   27   ports:
   28   - port: 80
   29     targetPort: 80
   30   selector:
   31     app: my-app

  ---


  Как это работает для вас (итог):
   1. Вы пишете код и делаете git push.
   2. GitLab запускает Build: собирает Docker-образ и отправляет его в Yandex Container Registry под уникальным тегом (хеш коммита).
   3. GitLab запускает Deploy: через установленный ранее Агент подключается к вашему кластеру K8s.
   4. Команда kubectl apply говорит кластеру: "Возьми новый образ из реестра и обнови приложение".
   5. Yandex Cloud обновляет поды (контейнеры) без остановки приложения.


  Советы для новичка:
   * Kaniko: Если docker-in-docker (dind) не работает или кажется сложным, посмотрите в сторону Kaniko — это инструмент для сборки образов внутри контейнера без прав root.
   * Helm: Когда манифестов станет много, изучите Helm. Это "менеджер пакетов" для Kubernetes, который упрощает управление деплоем.
   * Безопасность: Никогда не храните пароли и ключи в коде, только в GitLab Variables.

Чтобы запушить (отправить) готовый Docker-образ в Yandex Container Registry, нужно выполнить три основных шага: авторизоваться, правильно переименовать (тегировать) образ и отправить его.

  Вот подробная инструкция:


  1. Узнайте ID вашего реестра
  Зайдите в консоль Yandex Cloud, выберите сервис Container Registry и скопируйте ID вашего реестра (он выглядит примерно так: crp6fxxxxxxxxxxxxxxx).

  ---

  2. Авторизация (Docker Login)
  Есть три способа, выберите наиболее подходящий:


  Способ А: Через Yandex Cloud CLI (самый простой для локальной машины)
  Если у вас установлен и настроен yc, просто выполните:
   1 yc container registry configure-docker
  Эта команда один раз настроит ваш Docker так, чтобы он всегда "узнавал" облако Yandex.

  Способ Б: Через OAuth-токен (если нет CLI)
   1. Получите токен по ссылке: https://oauth.yandex.ru/verification_code (https://oauth.yandex.ru/verification_code)
   2. Выполните команду:


   1 echo "ваш_oauth_токен" | docker login --username oauth --password-stdin cr.yandex

  Способ В: Через статический ключ сервисного аккаунта (лучший для CI/CD)
  Если у вас есть JSON-файл ключа:


   1 cat key.json | docker login --username json_key --password-stdin cr.yandex

  ---


  3. Тегирование образа
  Docker должен понимать, в какое именно хранилище отправлять образ. Для этого имя образа должно строго соответствовать формату:
  cr.yandex/ID_РЕЕСТРА/ИМЯ_ОБРАЗА:ТЕГ

  Если ваш локальный образ называется my-app:v1, выполните:


   1 docker tag my-app:v1 cr.yandex/ваш_id_реестра/my-app:v1
  (Замените ваш_id_реестра на реальный ID из шага 1).

  ---

  4. Отправка образа (Push)
  Теперь просто отправьте его в облако:


   1 docker push cr.yandex/ваш_id_реестра/my-app:v1

  ---


  Как проверить результат?
  После завершения команды push:
   1. Вернитесь в консоль Yandex Cloud.
   2. Зайдите в ваш реестр.
   3. Там появится репозиторий my-app, а внутри него — ваш образ с тегом v1.


  Частая ошибка новичков:
  Ошибка: denied: Permission denied или unauthorized.
  Решение:
   * Проверьте, что вы правильно ввели ID реестра.
   * Убедитесь, что у вашего пользователя или сервисного аккаунта есть роль container-registry.images.pusher в облаке. Без этой роли загружать образы нельзя.

Эта функция называется GitLab Kubernetes Agent (KAS) User Access. Она позволяет вам использовать kubectl на вашем локальном компьютере, проксируя запросы через GitLab прямо в кластер, без необходимости открывать порты кластера наружу.


  Вот пошаговая инструкция, как это настроить и запустить:

  Шаг 1: Создание конфигурационного файла агента
  Агент настраивается через файл в вашем репозитории GitLab.


   1. В корне вашего проекта создайте папку (если её нет): .gitlab/agents/<имя-вашего-агента>/
       * Важно: <имя-вашего-агента> должно точно совпадать с тем именем, которое вы дали агенту при регистрации в GitLab.
   2. Внутри этой папки создайте файл config.yaml.
   3. Вставьте в него настройки доступа:


   1 user_access:
   2   access_as:
   3     agent: {} # Использует права сервисного аккаунта, под которым запущен агент (обычно cluster-admin)
   4   projects:
   5     - id: path/to/your/project # Полный путь к вашему проекту (например, my-group/my-app)

   4. Сделайте commit и push этого файла в основную ветку (main/master). Агент в кластере увидит изменения и обновится сам.

  ---


  Шаг 2: Настройка локального kubectl
  Теперь вам нужно "объяснить" вашему локальному kubectl, как подключаться к GitLab.


   1. Убедитесь, что у вас установлен kubectl.
   2. В интерфейсе GitLab перейдите в Operate -> Kubernetes clusters.
   3. Нажмите на имя вашего агента.
   4. Там будет вкладка "User Access" или кнопка "Connect". GitLab покажет вам готовую команду. Она выглядит примерно так:

   1 kubectl config use-context path/to/your/project:yc-agent


  Если команды нет, формат контекста всегда такой: полный/путь/к/проекту:имя-агента.

  ---

  Шаг 3: Проверка подключения
  Попробуйте выполнить любую команду:
   1 kubectl get pods


  Что произойдет в этот момент:
   1. kubectl отправит запрос на сервер GitLab (KAS).
   2. GitLab проверит ваши права (вы должны быть залогинены в GitLab через браузер или иметь валидный токен).
   3. GitLab перенаправит запрос агенту, который стоит внутри вашего кластера в Yandex Cloud.
   4. Агент выполнит команду и вернет результат вам в терминал.

  ---

  Почему это может не работать (решение проблем):


   1. Ошибка авторизации: Если вы используете это впервые, kubectl может попросить вас авторизоваться. Самый простой способ — установить GitLab CLI (glab) и выполнить glab auth login.
   2. Агент не в сети: Если вы до сих пор видите ошибку WebSocket dial..., значит, агент в кластере "мертв" или не может достучаться до GitLab. User Access не будет работать, пока сам агент не подключится к GitLab.
       * Проверьте еще раз NAT-шлюз в Yandex Cloud (нужен для исходящего трафика).
       * Проверьте логи пода агента: kubectl logs -n gitlab-agent -l app=gitlab-agent.
   3. Ветка по умолчанию: GitLab ищет файл config.yaml только в ветке по умолчанию (обычно main). Если вы создали файл в другой ветке, настройки не применятся.


  Совет: Для бесплатной версии GitLab (Free) доступен только режим access_as: agent: {}. Это значит, что любой пользователь с правами Developer в проекте получит в кластере те же права, что и сам агент (обычно полные права администратора). Будьте
  осторожны с этим.

можно поподробнее как создать переменную YC_SA_KEY: JSON-ключ вашего сервисного аккаунта. Какой имено текст вводит в поле значение

✦ Для того чтобы создать эту переменную, вам сначала нужно сгенерировать JSON-файл ключа в консоли Yandex Cloud, а затем скопировать его содержимое в GitLab.

  Вот пошаговая инструкция:

  Шаг 1: Создание ключа в Yandex Cloud (Консоль)


   1. Зайдите в консоль Yandex Cloud (https://console.cloud.yandex.ru/).
   2. Перейдите в каталог (Folder), где находится ваш сервисный аккаунт.
   3. В меню слева выберите Сервисные аккаунты.
   4. Нажмите на имя вашего сервисного аккаунта (у него должны быть роли container-registry.images.pusher и, желательно, k8s.clusters.agent).
   5. Нажмите кнопку Создать новый ключ -> Создать авторизованный ключ.
   6. В появившемся окне выберите тип JSON.
   7. Нажмите Создать. Браузер предложит вам скачать файл (например, authorized_key.json). Сохраните его.

  ---

  Шаг 2: Что именно копировать?


  Откройте скачанный файл любым текстовым редактором (Блокнот, VS Code и т.д.). Внутри вы увидите текст в формате JSON, который выглядит примерно так:


   1 {
   2    "id": "aje...",
   3    "service_account_id": "aje...",
   4    "created_at": "2023-10-27T...",
   5    "key_algorithm": "RSA_2048",
   6    "public_key": "-----BEGIN PUBLIC KEY-----\n...",
   7    "private_key": "-----BEGIN PRIVATE KEY-----\nMIIEvQIBADANBgkqhkiG9w0BAQEFAASCBKcwggSjAgEAAoIBAQ..."
   8 }

  Вам нужно скопировать ВЕСЬ этот текст целиком (включая все фигурные скобки { }).

  ---

  Шаг 3: Добавление переменной в GitLab


   1. Зайдите в ваш проект в GitLab.
   2. В меню слева перейдите в Settings -> CI/CD.
   3. Разверните раздел Variables и нажмите Add variable.
   4. В поле Key (Ключ) напишите: YC_SA_KEY
   5. В поле Value (Значение) вставьте весь текст из вашего JSON-файла, который вы скопировали на Шаге 2.
   6. Настройки (Flags):
       * Protect variable: Оставьте галочку, если хотите, чтобы ключ был доступен только в защищенных ветках (например, main).
       * Mask variable: Эту галочку для JSON-ключей обычно не ставят (она может выдать ошибку, так как формат JSON слишком сложный для маскировки).
   7. Нажмите Add variable.

  ---

  Как это используется в скрипте .gitlab-ci.yml?


  Когда в вашем скрипте встречается команда:
   1 echo "$YC_SA_KEY" | docker login --username json_key --password-stdin cr.yandex
  GitLab подставляет содержимое вашего JSON-файла вместо $YC_SA_KEY. Docker считывает этот JSON и понимает, что перед ним "паспорт" вашего сервисного аккаунта, после чего разрешает загрузку (push) образов в реестр.


  Важно: Никогда не показывайте этот JSON посторонним и не выкладывайте его в репозиторий, так как он дает полный доступ к ресурсам вашего сервисного аккаунта в облаке.

 как сделать доступным образ для того чтобы спулить лежащий в yandex registry

✦ Чтобы спуллить (скачать) образ из Yandex Container Registry, у вас есть три основных пути в зависимости от того, куда вы хотите его скачать: в Kubernetes, на локальную машину или сделать образ публичным.


  Способ 1: В кластер Managed Service for Kubernetes (Yandex Cloud)
  Это самый правильный и автоматизированный способ. Если ваш кластер K8s находится в том же облаке, что и реестр, вам не нужно создавать секреты вручную.


   1. Проверьте права сервисного аккаунта узлов (Nodes):
      У вашего кластера есть «Сервисный аккаунт узлов» (вы указывали его при создании кластера).
       * Зайдите в консоль YC -> IAM -> ваш сервисный аккаунт.
       * Убедитесь, что у него есть роль container-registry.images.puller на каталог или конкретный реестр.
   2. Проверьте привязку:
      Если роль есть, Kubernetes сам "подружится" с реестром. В манифесте deployment.yaml просто указывайте полный путь к образу:
      image: cr.yandex/ID_РЕЕСТРА/my-app:v1

  ---


  Способ 2: Через ImagePullSecrets (Если кластер "чужой" или нет прав)
  Если вы используете GitLab Agent или сторонний Kubernetes, вам нужно создать "ключ от двери" внутри K8s.


   1. Создайте секрет в Kubernetes (используя тот самый JSON-ключ key.json, который мы обсуждали ранее):


   1     kubectl create secret docker-registry yc-registry-secret \
   2       --docker-server=cr.yandex \
   3       --docker-username=json_key \
   4       --docker-password='$(cat key.json)' \
   5       --docker-email=my@email.com \
   6       -n default
   2. Укажите этот секрет в вашем Deployment:


   1     spec:
   2       containers:
   3       - name: my-app
   4         image: cr.yandex/ID_РЕЕСТРА/my-app:v1
   5       imagePullSecrets:
   6       - name: yc-registry-secret  # Имя секрета, который мы создали выше

  ---


  Способ 3: Сделать реестр публичным (Внимание: небезопасно!)
  Если вы хотите, чтобы любой человек или сервер в интернете мог скачать образ без всяких паролей:


   1. Зайдите в консоль Yandex Cloud -> Container Registry.
   2. Выберите ваш реестр.
   3. Перейдите на вкладку Access bindings (Права доступа).
   4. Нажмите Add role (Добавить роль).
   5. В поле "Select subject" выберите All users (или system:allUsers).
   6. Выберите роль container-registry.images.puller.
   7. Нажмите Save.

  Теперь образ можно будет скачать просто командой docker pull cr.yandex/ID_РЕЕСТРА/my-app:v1 без авторизации.

  ---

  Способ 4: На локальную машину
  Если вы хотите просто скачать образ себе на компьютер:


   1. Авторизуйтесь (через YC CLI):
   1     yc container registry configure-docker
   2. Спулльте образ:


   1     docker pull cr.yandex/ID_РЕЕСТРА/my-app:v1


  Что выбрать?
  Если вы настраиваете деплой в Yandex Kubernetes, Способ 1 — лучший. Просто дайте сервисному аккаунту роль puller в консоли облака, и всё заработает само.

 Предыдущий вопрос только кластер развернту на физических виртуальных машинах в yandex облаке а не через Managed Service for Kubernetes
▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄
✦ Если ваш кластер Kubernetes развернут на обычных виртуальных машинах (Self-hosted), а не через Managed Service, у него нет автоматической связи с реестром Yandex Cloud. Для Kubernetes эти виртуальные машины — просто "чужие" сервера.


  Чтобы кластер мог скачать образ, вам нужно использовать механизм ImagePullSecrets.

  Вот пошаговый план:


  Шаг 1: Создание секретного ключа (JSON)
  У вас уже должен быть JSON-файл ключа сервисного аккаунта (тот самый, который мы клали в переменную YC_SA_KEY в GitLab). Если нет — создайте его в консоли YC для сервисного аккаунта с ролью container-registry.images.puller.


  Шаг 2: Создание секрета в Kubernetes
  Вам нужно "положить" этот ключ внутрь вашего кластера Kubernetes, чтобы он знал пароль от реестра.

  Выполните команду на одной из ваших машин, где настроен kubectl:


   1 kubectl create secret docker-registry yc-registry-secret \
   2   --docker-server=cr.yandex \
   3   --docker-username=json_key \
   4   --docker-password='{ "id": "aje...", "service_account_id": "...", "private_key": "..." }' \
   5   --namespace default
   * yc-registry-secret — это имя секрета, которое мы потом укажем в настройках приложения.
   * --docker-password — вставьте сюда всё содержимое вашего JSON-файла (весь текст в фигурных скобках).
   * --namespace — укажите тот же namespace, в который вы будете деплоить приложение.

  ---

  Шаг 3: Указание секрета в Deployment
  Теперь в вашем файле deployment.yaml нужно явно сказать: "Для скачивания образа используй этот секрет".


  Добавьте блок imagePullSecrets в секцию spec вашего пода (на одном уровне с containers):


    1 apiVersion: apps/v1
    2 kind: Deployment
    3 metadata:
    4   name: my-app
    5 spec:
    6   replicas: 1
    7   template:
    8     spec:
    9       containers:
   10       - name: my-app
   11         image: cr.yandex/ваш-id-реестра/my-app:latest
   12       # --- ВОТ ЭТОТ БЛОК НУЖНО ДОБАВИТЬ ---
   13       imagePullSecrets:
   14       - name: yc-registry-secret # Имя секрета из Шага 2

  ---

  Шаг 4 (Автоматизация через GitLab CI)
  Чтобы не создавать секрет руками каждый раз, вы можете добавить его создание прямо в ваш .gitlab-ci.yml перед деплоем.

  Пример этапа деплоя:


    1 deploy:
    2   stage: deploy
    3   script:
    4     - kubectl config use-context path/to/project:agent-name
    5     # Создаем секрет (или обновляем его, если изменился), чтобы K8s мог пуллить образы
    6     - kubectl create secret docker-registry yc-registry-secret
    7         --docker-server=cr.yandex
    8         --docker-username=json_key
    9         --docker-password="$YC_SA_KEY"
   10         --dry-run=client -o yaml | kubectl apply -f -
   11     # Деплоим приложение
   12     - kubectl apply -f k8s/deployment.yaml


  Лайфхак: Как не прописывать секрет в каждом Deployment?
  Если вы не хотите добавлять imagePullSecrets в каждый файл, вы можете "привязать" его к стандартному сервисному аккаунту в нужном namespace:

   1 kubectl patch serviceaccount default -p '{"imagePullSecrets": [{"name": "yc-registry-secret"}]}'
  После этой команды все поды в этом namespace будут автоматически пытаться использовать этот секрет для скачивания образов.


  Почему это важно для физических VM:
  На Managed K8s за вас это делает облако. На своих VM вы сами отвечаете за авторизацию. Если вы этого не сделаете, вы получите ошибку:
  ErrImagePull или ImagePullBackOff со словами Failed to pull image: rpc error: code = Unknown desc = Error response from daemon: unauthorized.

как сделать deploy приложения из предыдущих шагов только когда добваляется tag версии (например, v1.0.0)
▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄▄
✦ Для того чтобы деплой срабатывал только при создании тега (например, v1.0.0), нам нужно использовать секцию rules в файле .gitlab-ci.yml.

  Также хорошей практикой будет использовать имя тега (v1.0.0) в качестве версии Docker-образа.


  Вот обновленный пример вашего .gitlab-ci.yml:


    1 stages:
    2   - build
    3   - deploy
    4
    5 variables:
    6   # Путь к вашему образу (замените ID_РЕЕСТРА на ваш)
    7   IMAGE_NAME: cr.yandex/ваш_id_реестра/my-app
    8
    9 build-image:
   10   stage: build
   11   image: docker:24.0.5
   12   services:
   13     - docker:24.0.5-dind
   14   rules:
   15     # Запускать сборку ТОЛЬКО если это тег, начинающийся на 'v'
   16     - if: $CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+/
   17   script:
   18     - echo "$YC_SA_KEY" | docker login --username json_key --password-stdin cr.yandex
   19     # Собираем образ с тегом из Git (например, v1.0.0)
   20     - docker build -t $IMAGE_NAME:$CI_COMMIT_TAG .
   21     - docker push $IMAGE_NAME:$CI_COMMIT_TAG
   22
   23 deploy-to-k8s:
   24   stage: deploy
   25   image:
   26     name: bitnami/kubectl:latest
   27     entrypoint: [""]
   28   rules:
   29     # Запускать деплой ТОЛЬКО если это тег
   30     - if: $CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+/
   31   script:
   32     # 1. Подключаемся к агенту
   33     - kubectl config use-context path/to/project:yc-agent
   34
   35     # 2. Создаем/обновляем секрет для пулла образа (для ваших физ. машин)
   36     - kubectl create secret docker-registry yc-registry-secret
   37         --docker-server=cr.yandex
   38         --docker-username=json_key
   39         --docker-password="$YC_SA_KEY"
   40         --dry-run=client -o yaml | kubectl apply -f -
   41
   42     # 3. Подставляем имя образа с НОВЫМ ТЕГОМ в манифест
   43     - sed -i "s|__IMAGE__|$IMAGE_NAME:$CI_COMMIT_TAG|g" k8s/deployment.yaml
   44
   45     # 4. Применяем изменения
   46     - kubectl apply -f k8s/

  Что изменилось:


   1. rules::
       * Это условие запуска. Выражение $CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+/ означает: "запускай эту задачу, только если в Git создан тег, который выглядит как v1.0.0, v2.1.5 и так далее".
       * Если вы просто сделаете git push обычного кода в ветку main, этот пайплайн не запустится.


   2. $CI_COMMIT_TAG:
       * Это встроенная переменная GitLab. В ней хранится имя вашего тега. Мы используем её для тегирования Docker-образа (docker build -t ...:$CI_COMMIT_TAG). Теперь в Yandex Registry у вас будут образы с именами v1.0.0, v1.0.1, а не просто хеши.


   3. Автоматическое создание секрета:
       * Я добавил команду kubectl create secret ... | kubectl apply -f -. Это гарантирует, что на ваших физических виртуалках всегда будет актуальный "пароль" для скачивания новых образов из реестра.

  ---

  Как теперь делать деплой (команды):

  Когда ваш код готов к выпуску:


   1. Зафиксируйте изменения:
   1     git add .
   2     git commit -m "Готов к релизу v1.0.0"
   2. Создайте тег:
   1     git tag v1.0.0
   3. Отправьте тег в GitLab:
   1     git push origin v1.0.0


  Что произойдет:
  GitLab увидит тег, поймет, что он соответствует правилу rules, и запустит сборку и деплой. Обычные пуши в main теперь будут игнорироваться (если вы не добавите для них отдельные правила).


  Совет:
  Если вы хотите, чтобы сборка (build) шла всегда, а деплой (deploy) — только по тегам, просто уберите блок rules из секции build-image. Тогда при каждом пуше вы будете знать, собирается ли ваш проект, но обновляться на сервере он будет только при
  создании версии.

✦ Это стандартный приём в GitLab CI для работы со специализированными Docker-образами. Давайте разберем каждую строку:


  1. image: name: bitnami/kubectl:latest
  В GitLab CI каждая задача (job) выполняется внутри Docker-контейнера.
   * Чтобы в скрипте работала команда kubectl, нам нужен образ, в котором эта программа уже установлена.
   * bitnami/kubectl — это популярный, легкий и безопасный образ от компании Bitnami, который содержит только саму утилиту kubectl.
   * Используя его, вам не нужно тратить время на установку kubectl (через apt-get install и т.д.) внутри скрипта. Она уже там есть.

  ---

  2. entrypoint: [""] — самое важное
  Это "техническое обнуление" настроек образа.


  В чем проблема:
  У многих специализированных Docker-образов (как kubectl, terraform, aws-cli) в настройках прописан так называемый Entrypoint. Это главная команда, которая запускается автоматически, как только контейнер включается.
  Например, в образе bitnami/kubectl по умолчанию прописан запуск самой программы kubectl.


  Как это мешает GitLab:
  GitLab Runner работает так: он запускает контейнер и пытается выполнить в нем свой собственный Shell-скрипт (те самые команды, которые вы написали в секции script:).
  Если entrypoint не обнулить, то вместо запуска скрипта GitLab, контейнер попытается выполнить kubectl <ваш_скрипт>, что приведет к ошибке, так как kubectl не поймет команды вашего скрипта.


  Что делает entrypoint: [""]:
  Эта строка стирает стандартную команду запуска образа.
   * Она говорит GitLab: "Запусти этот контейнер просто как пустую оболочку (shell), чтобы я мог выполнять в нем любые команды (включая kubectl, sed, echo и другие)".

  ---


  Итог простыми словами:
   * image: Выбираем "инструментальный ящик", где уже лежит молоток (kubectl).
   * entrypoint: [""]: Убираем автоматическую блокировку, чтобы мы могли сами решать, когда и как этим молотком пользоваться, а не чтобы он начинал "стучать" сам по себе при открытии ящика.


  Совет: Если вы используете какой-то образ в GitLab CI и задача падает с непонятной ошибкой сразу после старта — скорее всего, вам нужно добавить entrypoint: [""].

✦ Чтобы реализовать такую логику, мы воспользуемся возможностью GitLab CI переопределять переменные внутри правил (rules).

  Это самый изящный способ: мы создаем одну задачу на сборку, но меняем «имя» тега в зависимости от того, пуш это в ветку или создание тега.

  Вот как будет выглядеть ваш .gitlab-ci.yml:


    1 stages:
    2   - build
    3   - deploy
    4
    5 variables:
    6   IMAGE_NAME: cr.yandex/ваш_id_реестра/my-app
    7
    8 build-image:
    9   stage: build
   10   image: docker:24.0.5
   11   services:
   12     - docker:24.0.5-dind
   13   # Описываем логику выбора тега
   14   rules:
   15     # 1. Если это тег (например, v1.1.0)
   16     - if: $CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+/
   17       variables:
   18         FINAL_TAG: $CI_COMMIT_TAG
   19     # 2. Если это обычный пуш в ветку (не содержит тега v.X.X.X)
   20     - if: $CI_COMMIT_BRANCH
   21       variables:
   22         FINAL_TAG: $CI_COMMIT_SHORT_SHA
   23   script:
   24     - echo "$YC_SA_KEY" | docker login --username json_key --password-stdin cr.yandex
   25     # Используем переменную FINAL_TAG, которая определилась в rules
   26     - echo "Сборка образа с тегом: $FINAL_TAG"
   27     - docker build -t $IMAGE_NAME:$FINAL_TAG .
   28     - docker push $IMAGE_NAME:$FINAL_TAG
   29
   30 deploy-to-k8s:
   31   stage: deploy
   32   image:
   33     name: bitnami/kubectl:latest
   34     entrypoint: [""]
   35   rules:
   36     # Деплой запускаем ТОЛЬКО по тегам (как вы просили ранее)
   37     # Если хотите деплоить и ветки, удалите эту секцию или добавьте условие для веток
   38     - if: $CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+/
   39       variables:
   40         FINAL_TAG: $CI_COMMIT_TAG
   41   script:
   42     - kubectl config use-context path/to/project:yc-agent
   43
   44     # Создаем секрет для пулла
   45     - kubectl create secret docker-registry yc-registry-secret
   46         --docker-server=cr.yandex
   47         --docker-username=json_key
   48         --docker-password="$YC_SA_KEY"
   49         --dry-run=client -o yaml | kubectl apply -f -
   50
   51     # Подставляем тег (здесь он будет v1.1.0)
   52     - sed -i "s|__IMAGE__|$IMAGE_NAME:$FINAL_TAG|g" k8s/deployment.yaml
   53     - kubectl apply -f k8s/

  Разбор логики:


   1. Секция variables внутри rules:
       * Если GitLab видит, что вы создали тег (например, v1.1.0), он берет первое правило и присваивает переменной FINAL_TAG значение этого тега.
       * Если вы просто сделали git push в ветку main, срабатывает второе правило, и FINAL_TAG становится равен короткому хешу коммита (например, a1b2c3d4).

   2. Универсальный скрипт:
       * В самой команде docker build нам теперь не нужно писать условия if/else. Мы просто используем $FINAL_TAG, который GitLab уже заботливо подготовил для нас.


   3. Регулярное выражение ^v\d+\.\d+\.\d+:
       * Это защита. Она гарантирует, что правило сработает только на теги типа v1.0.0, v2.1.5. Если вы создадите тег test-tag, сработает правило для веток (и тег образа будет хешем), либо сборка вообще не запустится, если не попадет под условия.

  Как это будет работать на практике:


   * Вы запушили код в ветку: В Yandex Registry появится образ my-app:a1b2c3d4. Сборка прошла, но деплой (по моим правилам выше) не запустился.
   * Вы создали тег v1.1.0: В Yandex Registry появится образ my-app:v1.1.0. Сразу после сборки запустится деплой, который обновит ваш Kubernetes-кластер именно этой версией.


  Важный нюанс:
  Если вы хотите, чтобы деплой тоже работал для веток (с коротким хешем), просто скопируйте блок rules из build-image в deploy-to-k8s. Тогда каждое ваше изменение будет сразу улетать на сервер. Но обычно для серверов (Production) используют именно теги,
  чтобы контролировать выпуски.
